## Exercise-by-exercise feedback

### `w2_01_abs_difference_fn.rs`

The function is short, clear, and does not print. It correctly returns the absolute difference for ordinary `i32` values.

The remaining limitation is overflow. With extreme inputs, either `x - y` or `y - x` can exceed `i32::MAX`. For example, the mathematical distance from `i32::MIN` to `i32::MAX` cannot be represented by an `i32`.

This is partly imposed by the exercise signature. Do not build an elaborate fix yet, but recognize that “absolute difference of any two `i32` values returning `i32`” is not a total operation.

Didactic result: successful.

### `w2_02_grade_classifier_fn.rs`

The ascending threshold chain is correct and simpler than repeating both lower and upper bounds. Once a value reaches `n < 70`, for example, the previous branch has already established that it is at least 60.

Adding `N/A` for values above 100 is reasonable behavior in isolation, but it weakens the separation intended by the exercise. The next exercise introduces `is_valid_grade`, so now both the classifier and validator partially own validation.

The better conceptual boundary is:

- validation decides whether classification is allowed;
- classification assumes it receives a valid grade.

You do not need to rewrite it immediately. Just notice the duplicated responsibility.

### `w2_03_valid_grade_domain.rs`

`is_valid_grade` is exactly the right check for a `u8`: only the upper bound remains to be validated.

The loop tests all requested values, but invalid values produce no output. The Week 2 rules explicitly requested `invalid grade` for values above 100. Silence makes it harder to distinguish “correctly rejected” from “forgotten or skipped.”

This is the smallest unfinished piece in the grade exercise.

### `w2_04_caesar_encode_fn.rs`

The valid example encodes correctly. The function contains a clear modular transformation, and `main` validates the letter and reduces the key.

The function itself relies on its caller to provide a safe key. That is valid if its contract is strictly `letter: 0..25` and `key: 0..25`; under those constraints, `letter + key` is at most 50 and cannot overflow `u8`.

The name `SIZE` is understandable, but `ALPHABET_SIZE` would express the meaning more precisely without adding complexity.

Printing `-1` as an invalid sentinel is slightly misleading because the program otherwise works with `u8`, which cannot represent `-1`. It is only text, not a returned numeric value. At this stage, printing `invalid letter` would make the intent clearer.

### `w2_05_caesar_decode_fn.rs`

For a valid letter, this is safe with the way `main` calls it:

- adding 26 produces at most 51;
- the reduced key is at most 25;
- subtraction therefore cannot underflow.

The important weakness is that the safety rule lives outside `caesar_decode`. Its signature accepts any `u8`, but its body assumes the caller has already increased the letter and reduced the key. A function named `caesar_decode` should ideally own the complete decode calculation rather than require the caller to prepare a special intermediate value.

This is a responsibility-design issue, not an advanced Rust issue.

### `w2_06_distance_squared_fn.rs`

The function correctly decomposes the calculation into `dx`, `dy`, and squared distance. Reusing `abs_difference` demonstrates function composition.

Absolute differences are not necessary before squaring because a negative number and its positive counterpart have the same square. Still, using the earlier function is reasonable for this exercise.

Overflow remains possible in both the differences and the squaring. The function is correct only for a limited coordinate range that is not currently documented.

### `w2_07_closest_point_fn.rs`

This file contains the most important Week 2 correctness bug.

When `point_a > point_b`, point A has the larger squared distance and point B is closer. The function currently returns `"A"`. The next branch has the same reversal: when B is farther, it returns `"B"`.

The current `main` uses two points with equal squared distances, so it exercises only the correct `TIE` branch and hides the reversal.

This is a valuable example of why one passing case is insufficient. The function composition is good; the comparison interpretation is wrong.

Urgency: fix and manually check all three outcomes—A closer, B closer, and tie—before beginning the next study stage.

### `w2_08_array_transform_while.rs`

The program correctly preserves the source array, uses a second array, validates positions, and transforms elements with a bounded `while` loop.

The validation pass is thoughtful: it prevents a partially transformed result if an invalid letter appears later in the source.

There are two learning issues:

- printing occurs inside the transformation loop instead of after the calculation is complete;
- the array is named `letter`, singular, even though it contains several letters.

The first issue matters because Week 2's central rule is “calculation first, printing later.” The loop should finish constructing the result before presentation begins.

### `w2_09_array_sum_while.rs`

The accumulator loop is correct and safe for the current `[u32; 5]` values.

The exercise specifically requested a `[u8; 5]` source array and a `u32` accumulator. Using `[u32; 5]` skips the intended lesson: the source elements may be small while the accumulated result needs a wider type.

This is not merely a cosmetic restriction. It was designed to make you think about the type of each operand during accumulation.

### `w2_10_boundary_case_table.rs`

The boundary selection is strong and follows the exercise well. The `grader` function avoids duplicating the validity/classification decision at every call site.

It calls `is_valid_grade(n)` more than once for valid grades. That is harmless for this tiny pure function, but storing the boolean once would make the flow clearer.

The larger issue is structural: importing executable `.rs` files with `#[path] mod` also imports their private `main` functions as ordinary unused module functions. This explains the `dead_code` warnings recorded in `W2_RULES.md`.

The warnings do not mean your actual binary entry point is unused. They mean an earlier exercise file, including its own `main`, has been embedded as a module inside another binary.

## Correctness assessment

Most Week 2 examples are correct for their stated inputs. The exceptions or incomplete behaviors are:

1. `closest_point` returns the farther point instead of the closer one in non-tie cases.
2. `valid_grade_domain` silently skips invalid values instead of reporting them.

The first item is a direct logic bug. The others are domain or responsibility limitations.

## Safety assessment

There is no `unsafe` Rust and no visible memory-safety problem.

The `while` loops maintain valid array indices. The Caesar transformation validates source values before indexing or transforming them. The accumulator does not overflow for the current five small values.

Numeric overflow is still the relevant kind of safety concern. The geometry functions and signed absolute difference can overflow for extreme `i32` inputs. These functions need documented input limits before they can be described as correct for their entire type domain.

## Idiomatic Rust assessment

For your current stage, expression-based returns, small functions, typed arrays, and explicit loops are appropriate. Do not replace the loops with iterator chains just to appear idiomatic.

The non-idiomatic part worth addressing soon is the reuse structure:

```text
binary exercise file -> imported as module -> contains another main
```

This produces confusing warnings and makes every exercise file serve two roles.

The future Rust structure is:

```text
shared functions -> library/module file
binary main -> imports and demonstrates shared functions
```

Do not reorganize the whole project immediately. First understand why the warnings appear. A small shared module can be introduced when Week 3 or the next review explicitly focuses on modules.
